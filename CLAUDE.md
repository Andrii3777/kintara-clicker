# CLAUDE.md — Tree Bot (авто-кликер рубки деревьев)

Контекст проекта для новых сессий. Читать целиком перед изменениями.

---

## 1. Что это

Авто-кликер для браузерной изометрической игры **Kintara** (воксель-стиль,
вид сверху-сбоку). Бот находит деревья компьютерным зрением (OpenCV),
кликает по ним мышью (через WinAPI), ждёт срубки и переходит к следующему.
Когда деревья кончились — сам панорамирует карту и ищет новые.

Цель: фармить дерево (wood) автоматически.

**Стек:** Python 3.13, OpenCV (`cv2`), NumPy, `mss` (захват экрана),
`keyboard` (стоп по 'q'), `ctypes`/WinAPI (клик и DPI). `pyautogui` в
requirements есть, но в боте НЕ используется (заменён на `win_input.py`).

**Платформа:** Windows 11. Запуск из Git Bash / PowerShell.

---

## 2. Специфика игры (ВАЖНО — на этом построена вся логика)

1. **Изометрия, воксель-блоки.** Дерево = тёмно-зелёная крона-куб + тонкий
   коричневый ствол на траве. Газон — тоже зелёный (терен из зелёных блоков),
   поэтому отличить дерево от фона по одному цвету нельзя.
2. **Камера СТАТИЧНА.** Не едет за персонажем. Перс ходит по экрану, объекты
   остаются на фикс-позициях. → следим за глоу в точке клика дерева, а карту
   двигаем сами перетягиванием.
3. **Клик по дереву = перс идёт к нему и рубит.** Дистанция разная → время
   ходьбы разное. Нельзя ждать фикс-время.
4. **При клике дерево ПОДСВЕЧИВАЕТСЯ** ярким салатовым глоу (+ всплывает «+1»).
   Глоу появляется когда перс ДОШЁЛ и начал рубить (НЕ в момент клика!),
   **горит весь руб**, гаснет когда дерево срублено. Это главный сигнал «готово».
5. **Карта больше экрана.** Видно только кусок. Чтобы открыть другой —
   зажать ЛКМ на карте, протянуть, отпустить (drag-to-pan).
6. **Край карты** = фиолетовая пустота (off-map). Карта — ромб (изометрия),
   пустота в углах.
7. **Мультиплеер.** На экране бывают другие игроки (с никами), их можно спутать
   с объектами. Кусты/декор/дорожки/дома похожи по цвету на деревья.

## 4. Как работает детекция деревьев (`tree_detector.py`)

Пайплайн в `detect_trees(bgr)`:

1. **BGR → HSV.** Цвет в HSV ловится диапазоном.
2. **Три маски** (`_build_masks`):
   - `green` (КРОНА) — тёмный зелёный `H 32-90, S 60-255, V 40-150`.
     Ключевой инсайт: **газон яркий (V~169), крона тёмная (V~117-153)**.
     Берём только тёмный зелёный → газон выпадает, крона остаётся.
   - `brown` (СТВОЛ) — `H 22-42, S 150-255, V 60-150`. Замеры: ствол
     `H27-39 S180-202`, дом `H17-18 S77-92`. **Насыщенность S>150 отсекает
     дома/дороги** (у них S низкая). Морфология: ствол тонкий (1-2px), поэтому
     **только CLOSE+DILATE, БЕЗ OPEN** (open стирал стволы в ноль — был баг).
   - `grass` (широкий зелёный) — для скана основания ствола (если нужно).
3. **Контуры коричневого** (стволы). Для каждого:
   - Фильтр площади: `MIN_TRUNK_AREA=8 .. MAX_TRUNK_AREA=4000`, `MAX_TRUNK_WIDTH=60`.
     Дома/дороги/брёвна — большие → выкид.
   - **UI-зоны** (`IGNORE_ZONES`, доли экрана): низ (хотбар), чат слева-снизу,
     миникарта справа-сверху → выкид.
   - **Проверка кроны над стволом:** в зоне выше ствола доля тёмной кроны
     должна быть `>= CROWN_MIN_RATIO=0.30` И абсолютно `>= CROWN_MIN_AREA=60`.
     Это отсекает забор/брёвна/голую траву (над ними нет тёмной кроны).
4. **Точка клика** (`CLICK_TARGET`):
   - `'crown'` (по умолч) — центр масс кроны. **Игра принимает клик по кроне**,
     это надёжнее основания (не зависит от тонкого стыка).
   - `'base'` — скан вниз от ствола до травы (стык ствол↔трава).

Возврат: список деревьев `{box, click, crown_ratio, crown_area, trunk_area}`
плюс `(rejected, green, brown)` для debug.

**Доп. функции в детекторе:**
- `highlight_count(bgr, cx, cy, r)` — кол-во пикселей ЯРКОЙ подсветки
  (`HIGHLIGHT H 35-90, S 40-255, V 195-255`) в зоне. Глоу рубки → сотни px,
  трава/перс → ~0. Замеры: глоу `V206-253`, на дереве r40 ~2600px, трава 0.

## 5. Как работает рубка и ожидание (`bot.py`)

Главный цикл `main()`:

1. Захват монитора `MONITOR_INDEX`.
2. `detect_trees` → `pick_confident` (ствол ≥`MIN_CLICK_TRUNK_AREA=12`,
   крона ≥`MIN_CLICK_CROWN_AREA=80` — только явные деревья).
3. **Выбор цели** `choose_target`:
   - первое дерево — ближайшее к `CHAR_ANCHOR` (перс ~центр).
   - дальше — ПРОСТО ближайшее к `last_pos` (прошлое дерево). Без отсечки по
     дистанции: срубленное исключено через `recent`. `SAME_TREE_R=18` — радиус
     дедупа «то же дерево» между кадрами (джиттер кроны), МАЛЕНЬКИЙ чтоб не съесть
     соседа впритык (был баг: `SKIP_RADIUS=30` гнал бота к дальнему вместо близкого).
   - **мягкий кулдаун** `recent` (`COOLDOWN_SEC=12`): не кликать недавнюю точку
     (антизацикливание), НО можно вернуться если ничего больше нет.
   - **жёсткий чёрный список** `blacklist` (`BLACKLIST_TTL=45`): дерево давшее
     `no_highlight` (куст/декор/недоступно) — НЕ кликаем, без фолбэка. Если
     всё в чёрном списке → `target=None` → пан.
4. **Клик** `win_input.click_abs` (координаты + смещение монитора).
5. **Ждём срубку** `wait_until_chopped(tree_px, dist)` — по ГЛОУ в точке дерева.
   Глоу меряем ОТНОСИТЕЛЬНО `base` = глоу в МОМЕНТ клика (перс не дошёл, рубки
   нет). Абсолютные пороги врали в густом лесу (яркий фон → всё в `timeout`):
   - **Фаза 1 (появление):** ждём пока глоу `>= base + NEON_RISE_PX=180`.
     Бюджет = `APPEAR_BASE=3 + dist*WALK_SEC_PER_PX=0.02`, потолок `APPEAR_MAX=6`.
     **Растёт с дистанцией — НЕ бросаем цель пока перс идёт.**
     Глоу не появился → `no_highlight` → в чёрный список.
   - **Фаза 2 (погасание):** ждём пока глоу `<= base + NEON_FALL_PX=60` подряд
     `VANISH_CONFIRM=2` раз → `chopped`.
   - **`HARD_CAP=8`** — жёсткий потолок на всё: завис → `timeout` → дальше.
   - Возвращает `(status, (base, peak, last))` — лог печатает числа глоу для
     калибровки (`-> chopped за 3.2s (глоу base=… peak=… last=…)`).
6. Пауза `POST_CHOP_PAUSE` (0.1-0.3с), следующее дерево.

**Почему по глоу, а не по исчезновению кроны:** при клике крона становится
ярким глоу → ломала старые сигналы «появилась трава». Глоу — чистый,
однозначный сигнал состояния рубки. Подтверждено: «держится весь руб».

## 6. Панорамирование карты (`common/pan.py`, `Panner`) — ОБЩЕЕ

Логика обхода **общая** для всех ботов (`common/pan.py`, класс `Panner`) — раньше
дублировалась 4 раза. Бот только включает флагом и передаёт регион + параметры.

**ВЫКЛ по умолчанию.** `PAN_ENABLED = False` в CONFIG каждого бота. Целей нет →
бот просто **ждёт на месте** (single: `time.sleep(NO_*_WAIT)`; x4: `resume_at=now+1`),
карту НЕ двигает. `True` → обход «газонокосилкой».

Целей нет ИЛИ все в чёрном списке (и `PAN_ENABLED=True`) → обход по **ДИАГОНАЛЯМ
ромба** (изометрия — оси карты идут ↗↙ и ↘↖, не горизонт/вертикаль):

```
ряд вдоль оси SW↔NE (PAN_ROW_LEN панов) → шаг по оси SE↔NW → обратный ряд → …
после PAN_NUM_ROWS шагов → разворот оси шага. Всё по СЧЁТЧИКАМ (детект края убран).
```

- Всё по счётчикам, без детекта края: у края игра «пружинит» (счётчик тикает,
  карта стоит) → детект ненадёжен. Заезд в пустоту безвреден (целей там нет).
- Потолок `max_empty = 2*PAN_NUM_ROWS*(PAN_ROW_LEN+1)+8` Panner считает сам →
  `step()` возвращает `True` = карта обойдена (single: стоп, x4: `state=DONE`).
- `Panner.reset()` после перезахода (карта/координаты другие) и при появлении цели.
- Сброс фарм-памяти (`last_pos`/`recent`) и FSM — на стороне БОТА, не Panner.

**API:** `Panner(region, cfg, dry_run, tag, grab, log)`; `step(reason, before)` →
bool; `reset()`. `region` = dict `left/top/width/height` (монитор для single,
квадрант для x4) — drag из его ЦЕНТРА. `grab` (опц.) меряет change% для лога.

Drag: `win_input.drag_abs` — зажать ЛКМ → плавно вести 25 шагов за `PAN_DURATION`
→ отпустить. Плавность чтоб игра распознала drag, не клик.

**Zoom после входа на сервер (`zoom_map`, ОБЩЕЕ).** Зашли в мир (UI/перезаход
закончился) → один раз крутим колесо `ZOOM_TIMES=6` раз в центре региона
(`ZOOM_NOTCHES=-1`, вниз = zoom out; наоборот → поставь `>0`), карта уменьшается,
дальше фарм. Флаг `ZOOM_ON_ENTER` (по умолч ВКЛ). Триггер в боте: `was_in_ui`
(True на старте и при любом UI-действии) → при первом кадре без UI зумим и сбрасываем
фарм-память/`panner`. Срабатывает и при первом запуске, и после каждого перезахода.

## 7. Клик и мультимонитор (`win_input.py`)

Проблема: `pyautogui` врёт на втором мониторе (отриц. координаты `left=-1920`)
и при DPI-масштабе ≠100%.

Решение:
- `set_dpi_aware()` — процесс DPI-aware (Per-Monitor v2). **Вызывается ПЕРВЫМ**
  в `bot.py`, до `mss`/любого захвата (иначе mss ставит system-DPI по первому
  монитору, второй мажет).
- `click_abs(x,y)` / `drag_abs(...)` — ctypes `SendInput` с `MOUSEEVENTF_ABSOLUTE
  | VIRTUALDESK`, координаты нормированы по ВСЕМУ виртуальному столу (0..65535).
  Отрицательные координаты второго монитора работают корректно.

Координаты везде — ФИЗИЧЕСКИЕ пиксели виртуального стола = `cx + mon['left']`,
`cy + mon['top']` (mon из `mss`).

## 8. Запуск и калибровка

**Установка:** `pip install -r requirements.txt`
(`keyboard` иногда требует терминал от админа для глобальной 'q').

**Порядок** (все команды из папки `tree/`: `cd tree`):
1. Узнать монитор с игрой → выставить `MONITOR_INDEX` в `bot.py`.
2. Проверить детект: `python debug_detect.py --screen --monitor N` → смотреть
   `debug_result.png` (зелёные рамки = деревья, точки = клик; красные = отклонено).
   Маски `debug_green_mask.png`/`debug_brown_mask.png` — для подбора HSV.
3. Тест перетягивания: `python test_drag.py` (before/after, % изменения).
4. Сухой прогон: `DRY_RUN=True` в `bot.py` → `python bot.py` (печать без кликов).
5. Бой: `DRY_RUN=False` → `python bot.py`. Стоп — **q**.

**Диагностика глоу** (если рубка определяется криво):
`python diag_glow.py` — кликает дерево, 20с логирует пиксели глоу во времени.

**HSV/пороги калибровались на скрине 2560×1600.** На другом разрешении/яркости
проверять на своём захвате.

## 9. Полный список крутилок (`bot.py` CONFIG)

**Рубка/ожидание:**
- `TREE_GLOW_R=40` — радиус зоны глоу у дерева.
- `NEON_RISE_PX=180` / `NEON_FALL_PX=60` — пороги «горит/погас» ОТНОСИТЕЛЬНО
  `base` (фон в момент клика). Калибруй по числам в логе (`глоу base/peak/last`).
- `APPEAR_BASE=3`, `WALK_SEC_PER_PX=0.02`, `APPEAR_MAX=6` — бюджет ожидания
  прихода перса (растёт с дистанцией). Бросает цель раньше времени → подними
  `WALK_SEC_PER_PX`/`APPEAR_MAX`.
- `HARD_CAP=8` — жёсткий потолок одного дерева. Завис → через столько дальше.
- `VANISH_CONFIRM=2`, `CHOP_POLL=0.2`, `POST_CHOP_PAUSE=(0.1,0.3)`.

**Выбор цели:**
- `MIN_CLICK_TRUNK_AREA=12`, `MIN_CLICK_CROWN_AREA=80` — строгость «уверенного» дерева.
- `CHAR_ANCHOR=(0.42,0.40)` — где перс для выбора первого дерева.
- `SAME_TREE_R=18` (дедуп «то же дерево»), `COOLDOWN_SEC=12`, `BLACKLIST_TTL=45`.

**Пан:**
- `PAN_DIST=500`, `PAN_DURATION=0.6`, `PAN_SETTLE=0.6`.
- `PAN_ROW_LEN=6` — панов в ряду (карта шире → больше). `EDGE_CHANGE_FRAC=0.10`.
- `MAX_EMPTY_PANS=50`, `PAN_START_H='left'`, `PAN_START_V='down'`.

**Детект** (`tree_detector.py`): `GREEN_*`, `BROWN_*`, `HIGHLIGHT_*` диапазоны,
`CROWN_MIN_RATIO/AREA`, `IGNORE_ZONES`, `CLICK_TARGET`.

## 10. История решений (почему так, а не иначе)

- Шаблон (`matchTemplate`) **отвергнут** — деревьев много видов, изометрия,
  не масштаб-инвариантно. Пошли по цвету/форме.
- Клик: сначала основание ствола → потом выяснили **клик по кроне работает** и
  проще (`CLICK_TARGET='crown'`).
- Ожидание сруба: фикс-время → исчезновение кроны → **подсветка (глоу)**.
  Каждый шаг ловил баг (загораживание персом, глоу=яркий зелёный путал «траву»,
  камера). Финал — глоу в точке дерева + бюджет ожидания от дистанции.
- Пан: цикл направлений → пан-от-края по пустоте (осцилляция в углах ромба) →
  **счётчик панов в ряд** (надёжная вертикаль, не зависит от детекта края).

## 13. Перезаход на сервер (`common/reconnect.py`) — ВСЕ боты

Игра вылетает / кидает в очередь / показывает меню выбора сервера. Бот ловит это
и сам возвращается в мир. **Высший приоритет — выше фарма.**

**Разделение:** `reconnect.py` — ТОЛЬКО зрение (как `tree_detector`: вход кадр,
выход решение). Захват экрана и мышь — на стороне бота (`win_input`). Так общий
модуль не знает про mss/монитор/квадрант.

**OCR, НЕ matchTemplate.** Раньше ловили UI по пиксель-точным шаблонам
`common/ui/*.png` мульти-скейлом — костыль на ЧУЖОМ разрешении, ломался на
другом экране. Теперь читаем **текст кнопок** через RapidOCR (`rapidocr_onnxruntime`,
onnxruntime, без torch). Масштаб/разрешение/монитор-независимо, `*.png` больше
НЕ нужны (`load_templates(ui_dir)` игнорирует `ui_dir`, возвращает `{'_ocr':engine}`
ради совместимости со старым контрактом ботов — `.items()`/missing-check). Движок —
ленивый синглтон; нет пакета → перезаход тихо выключается, фарм живёт.

**Кадр кормить ЦВЕТНОЙ (BGR), не gray** — в gray OCR теряет текст на цветных
кнопках (проверено: `OK` читался в color, исчезал в gray). Боты передают `bgr`.

**РЕАЛЬНЫЙ ФЛОУ ИГРЫ** (по debug-скринам `sreens to debug/`):
1. **Список миров** — заголовок `SELECT A SERVER`/`CHOOSE YOUR REALM`. Скроллить.
   - `KINTARA CLUB N` + `MEMBERS ONLY`/`SUBSCRIBE` = **платный → пропустить**
     (у него нет строки `IN QUEUE`/`INSTANTLY` → отсеивается сам).
   - `SERVER N` = **бесплатный**: `OPEN · JOIN INSTANTLY` (приоритет 1, заходим
     сразу) либо `FULL · X IN QUEUE` (берём наименьший X если открытых нет).
2. Выбрали → `YOU ARE IN QUEUE` (`players ahead`) → **ждём**, ничего не жмём.
3. В очереди может выскочить `SOMETHING WENT WRONG`/`CONNECTION ERROR` → жмём
   `RETRY` → заново выбираем сервер.

**Логика** `emergency_ui_check(frame, target_server=None, templates, state)` —
приоритет сверху вниз, первый сработавший dict или `None` (`MIN_CONF=0.5`):
1. **RETRY/ошибка** (`RETRY`/`WENT WRONG`/`CONNECTION ERROR`) → клик RETRY. **ВЫШЕ
   очереди** — текст ошибки содержит «queue from another tab», иначе сработала бы 2.
2. **очередь** (`PLAYERS AHEAD`/`YOU ARE IN`/`ENTER AUTOMATICALLY` — спец-фразы, НЕ
   голый `QUEUE` который есть в списке `X IN QUEUE`) → `{action:'wait'}`.
3. `LOADING`/`CONNECTING`/`STARTING` → wait.
4. `OK` (точный токен) → клик. Покрывает попап вылета И «Server Announcement»
   (server update in 60s) — обе кнопки = `OK`.
4b. `TRAVELS` → клик САМОГО НИЖНЕГО бокса (`_find_lowest`) — попап торговца
   «Safe travels.»: фраза есть и в теле текста, и на кнопке → кнопка внизу.
5. `PLAY`+`NOW` кросс-бокс → клик Play Now (главное меню).
6. **выбор сервера** (заголовок ИЛИ распознаны строки серверов) → `_server_select`.

`target_server` больше **НЕ используется** (выбор динамический по очереди/instant),
оставлен в сигнатуре и в CONFIG ботов для совместимости.

**`_server_select(items, state)`** — нужна ПАМЯТЬ между кадрами:
- `_parse_servers`: строки с `INSTANTLY` (q=0) или числом из `(\d+) IN QUEUE`.
- Виден `JOIN INSTANTLY` → клик сразу (`state.clear()`).
- Иначе скроллим список. **Детект низа**: сигнатура очередей не сменилась после
  скролла ИЛИ `scrolls>=MAX_SCROLLS=15` (страховка от джиттера OCR) → коммит
  наименьшей очереди. Память (`prev_sig`,`scrolls`) — в `state`.
- scroll-действие несёт `focus_x/focus_y` (центр заголовка — **безопасная** точка
  фокуса окна, НЕ карточка!) + `x/y` (центр видимых строк — зона колеса). Без них
  фокус-клик попал бы по карточке = случайный выбор сервера.

**Зоны-мусор (`IGNORE_ZONES`):** OCR ловит текст вне попапов — верх БРАУЗЕРА
(адресная строка `kintara.gg/play`→`PLAY`), чат снизу-слева (`donate now`→`NOW`),
таскбар. Это давало ложный «Play Now» (PLAY из URL + NOW из чата → клик по адрес-
строке). Попапы ВСЕГДА по центру → боксы из этих зон выкидываем до анализа.
Доли зон подгоняются под высоту хрома браузера.

**Анти-ложь:** на живом экране OCR читает чат/ники (`PLAYER1`→`PLAY`). Многословные
кнопки требуем кросс-бокс целиком (`_find_multi`), `OK` — точным токеном, очередь —
спец-фразами. Проверено: игровой кадр с чатом → `None`. Весь флоу прогнан на
`sreens to debug/` — все переходы верны.

Координаты в dict — **локальные внутри региона**; бот добавляет offset.

**Интеграция в боте** (одиночные `*/bot.py` и x4):
- `load_templates()` раз при старте (создаёт OCR ~1-2с). Нет OCR → предупреждение,
  фарм живёт.
- Каждый аккаунт держит **`ui_state={}`** и передаёт в `emergency_ui_check`. В x4
  `_reset_farm()` его НЕ трогает (память скролла ведёт сам reconnect).
- Троттл `next_ui` — не сканить UI каждый кадр (OCR ~50-300мс/кадр).
- **UI-предчек В НАЧАЛЕ цикла**, после захвата, ДО детекта.
- `scroll`: фокус-клик по `focus_x/focus_y` (заголовок) → колесо в `x/y` (список).
  Старые `UI_SCROLL_FOCUS_DY/INTO_DY` остались фолбэком если focus_* нет в action.

## 14. Режим x4 — 4 аккаунта (`tree-4x/`, `stone-coal-4x/`)

4 окна игры «плиткой» 2×2 на одном мониторе, каждый квадрант = свой логин/сервер.

**Ключевой принцип — интерливинг, НЕ второй курсор.** Курсор в системе ОДИН
(SendInput, абс. координаты). 4 бота не двигают его разом. Но бот почти всё время
**ждёт** (перс идёт + добывает, до `HARD_CAP` сек). Пока акк-0 ждёт глоу — курсор
свободен, кликаем акк-1/2/3. ~4× за счёт интерливинга пауз.

**Архитектура:**
- Каждый аккаунт = **конечный автомат** (`AccountBot`): `SCAN → WAIT_APPEAR →
  WAIT_VANISH → (chopped) → SCAN`, либо `→ PAN` если целей нет, либо `DONE`
  (обошёл карту, `empty_pans > MAX_EMPTY_PANS`).
- **Планировщик** (`main`): по кругу даёт каждому аккаунту ОДИН `tick()` —
  максимум один захват региона + максимум одно действие мышью (клик ИЛИ один шаг
  пана). Блокирующего ожидания нет → аккаунты не держат друг друга.
- **Все дедлайны по `time.time()`** (не по итерациям) → интерливинг их не ломает.
- Троттл `last_poll` на аккаунт (≥`CHOP_POLL` между поллами одного), но другие
  тикают свободно. `time.sleep(0.01)` в конце круга — лёгкий yield.

**Регионы** (`build_regions`): авто-2×2 квадранты `MONITOR_INDEX`, координаты
АБСОЛЮТНЫЕ (left/top со смещением монитора). Флаги `QUADRANT_TL/TR/BL/BR` —
включить/выключить аккаунты (можно гонять 2 или 3). `REGION_INSET` — поджать
рамку внутрь если по краю квадранта мусор. `grab_region` / `_abs` переводят
локальные точки в абсолютные.

**Перезаход (§13) per-quadrant:** каждый `AccountBot` хранит свой
`target_server` (`QUADRANT_*_SERVER`) и `next_ui`. `_tick_ui` в начале `tick()`
(высший приоритет) → попап → действие + `_reset_farm()`.

**Калибровка x4:** квадрант = ¼ экрана → объекты МЕЛЬЧЕ. Пороги площади
(`MIN_CLICK_TRUNK_AREA`/`MIN_CLICK_CROWN_AREA`) возможно уменьшить против
одиночного бота — проверять на debug-кадре квадранта. Пан — общий `Panner`
(common/pan.py, см. §6): каждый аккаунт держит свой `self.panner` (drag из центра
СВОЕГО квадранта), `PAN_ENABLED` по умолчанию ВЫКЛ.

**Запуск:** `cd tree-4x` (или `stone-coal-4x`) → `python bot.py`. Разложить
4 окна по квадрантам за 3 сек старта. `DRY_RUN=True` — FSM крутится на
фикс-паузах без кликов. Стоп — `q`.
